Web组成

    Web访问过程
		1.客户端用户在浏览器中输入某个URL，eg:www.baidu.com
		2.客户端操作系统做DNS解析，获得目标服务器IP地址
		3.客户端操作系统打开一个自由端口，向服务器发起连接请求.
		4.通过三次握手,服务器端确认与该客户端的连接，并打开一个自由端口与该客户端通信
		5.客户端开始发送HTTP请求数据(以4KB为一个包，挨个传送)->发送HTTP Request
		6.服务器开始接收请求数据包，接收完成后，开始处理接收到的数据->Request Handling
		7.服务器开始返回响应数据(以4KB为一个包，挨个传送)->返回HTTP Response
		8.响应结束后，服务器会经过四次握手,断开连接，关闭各自的自由端口
	
	HTTP(Hyper Text Transform Protocol)
		它是TCP/IP协议族(栈)中"应用层"的一个协议,用于服务器和客户端间传输超文本内容(HTML、CSS、js、图片、音视频...)默认端口80

		历史
			HTTP/0.9->91年制定，有严重设计缺陷，只支持GET方法，不支持MIME类型
				MIME->Multipurpose Internet Mail Extensions多用途互联网邮件扩展类型,早期用于在电子邮件中指定附件的文件类型的扩展表达方式;
				image/jpeg		.jpg/.jpeg
				text/html 		.htm/.html/.xhtml
				application/json	.json
				application/xml		.xml
				知道MIME类型，客户端就知道调用什么解析器来解析对应文件
			HTTP/1.0支持多种请求方法，支持MIME
			HTTP/1.0+支持久连接、虚拟主机、代理连接
			HTTP/1.1校正HTTP设计中的缺陷，性能优化，删除不好特性(主流)
			HTTP-NG(HTTP/2.0)关注性能优化以及更强大的服务逻辑远程执行框架

		HTTP1.0 vs HTTP1.1
			1.0会在响应结束后，立即断开连接,以保证服务器可以连接多个客户端,这样的缺点可能会造成多次请求服务器.eg:请求一篇HTML文档,发现文档里还有CSS和图片(此时和服务器已断开连接)，使用1.0会再次请求服务器;所以1.0是offline模式

			1.1可以设置Connection:Keep-Alive让客户端和服务器保持连接,在响应后，会等待一段时间(具体时间，服务器配置),若无请求，则断开。若还有请求，则直接发送请求资源(无需再经过建立连接，请求数据过程);并支持虚拟主机和代理连接等特性

		URL
			URI->Unified Resource Identifier统一资源识别符->URI包含URL和URN
			URL->Unified Resource Locator 统一资源定位符->http://www.baidu.com/q?kw=xxx
			URN->Unified Resource Naming统一资源命名符->mailto:admin@xx.com

			URL完整格式
			
			
				scheme->协议,不区分大小写
				js中encodeURI()不会对://@?;#和英文编码
				encodeURIComponent()会对所有字符进行编码
				参数->采用"key=value"形式,有多个时,用分号分隔

		HTTP通信
			Message->消息/报文,HTTP客户端和服务器间传递的数据块，消息必须符合特定的格式才能被彼此理解
				Request Message->请求消息,客户端向服务器发送的请求消息
				Response Message->响应消息,服务器端根据客户端的请求消息，返回给客户端的响应消息

			Message基本结构(格式)
				每个消息都由三部分组成
					Start Line:消息起始行，必须，消息的基本描述信息
					Header:消息报头/头部,可能有0~N个,消息详细属性
					Body:消息主体,可选,包含数据的主体,其中数据可以为空，字符数据(HTML、CSS、JS等字符数据)或者二进制流数据(图片、音频、视频等字节数据)

					***消息头和起始行都是纯ASCII字符(不能有中文字符),主体中可有字符也可有字节;报文每行必须以CRLF(回车换行)结束***

					***报头和主体之间必须有一个空行(CRLF),若无主体，则报头之后也必须跟一个空行***

			Request Message具体格式
				1.起始行/请求行
					格式:请求方法 空格 请求URL 空格 协议版本CRLF

					请求方法:指定了客户端想对指定的资源/服务器做何种操作;可用的请求方法有
						GET	指明客户端想从服务器端获取指定的资源
						POST	指明客户端想发送给服务器一些数据
						PUT	指明客户端想让服务器保存某个资源(用的少)
						DELETE	指明客户端想让服务器端删除某个资源(用的少)
						HEAD	指明客户端只想查看指定资源的响应头部信息，而不要资源本身(一般用在需要缓存，比对文件是否被修改过的时候)
						TRACE	客户端可以对请求消息的传输路径进行追踪(用的少)
						OPTIONS	客户端询问服务器可以提交哪些请求方法(用的少)

						***除了POST和PUT请求方法外，其他请求(特别注意GET无请求主体)皆无请求主体***

						最常见的请求方法
							GET:浏览器地址栏中直接输入URL、页面内容解析时引入外部资源、超链接跳转、js跳转、GET方式提交表单等都是GET请求
							POST:POST方式提交表单的时候是POST请求

							***GET会将请求明文放在URL中发送给服务器,POST则是将请求内容明文放在请求主体中发给服务器，相对GET安全一点点***

							***在form提交表单的时候,若action中地址写的是2.html?pno=3,用GET方法提交时，提交的数据会将pno给覆盖掉(不会提交pno,只会提交form表单中的数据),而POST不仅会将form中数据提交给服务器还会将pno也提交给服务器***

						***上传文件必须用POST,不能用PUT并且form标签必须指明enctype为mulitpart/form-data***
						

				2.消息头部/请求头部
				 	常用请求头(描述请求本身、客户端、请求主体的特征)
				 		通用头部->在请求或响应头中都可以使用
				 			Connection:keep-alive	告诉对方启用持久连接
				 			Date: Wen 25 April 2015, 12:22:25 GMT 	告诉对方发出此次消息的时间
				 			Cache-Control: no-cache / max-age=3600   告诉对方如何缓存当前消息给的数据
				 			Pragma: no-cache  告诉对方如何缓存当前消息给的数据(适用于老版本的服务器和浏览器)

				 		请求专用头->只能出现在请求头中
				 			Host: www.baidu.com		适用于支持虚拟主机的服务器，客户端告诉服务器自己要访问的是哪一个虚拟主机
							Referer：http://tedu.cn/1.html  告诉服务器当前请求基于哪个页面发起的
							User-Agent: Mozilla....		告诉服务器客户端浏览器的类型
							Client-IP:128.1.5.4		告诉服务器客户端自己的IP地址
							Accept: text/html,application/javascript,*/*  告诉服务器客户端可以接受哪些响应内容类型
							Cookie: username=tom

				 		请求主体可用头部->描述请求主体特征
				 			Content-Type: text/plain(很少用)、application/x-www-form-urlencoded(表单提交)、multipart/form-data(文件上传)三者之一，告诉服务器请求主体的内容类型
							Content-Length: 5086  				告诉服务器请求主体的内容长度
							Expires: Wed 25 April 2015,12:22:25 GMT		告诉服务器当前的请求主体的有效期

				 		扩展头部->自定义头部(很少用)

				3.消息主体/请求正文
					保存着客户端提交给服务器，需要服务器加以处理(POST)或者保存(PUT)的数据;
					请求主体可以包含字符数据或字节数据
			Response Message具体格式
				1.起始行/响应行
					协议版本 空格 状态码 空格 状态说明CRLF
				2.消息头部/响应头部
					常用响应头(描述服务器信息)
						通用头部：Connection / Date / Cache-Control / Pragma

						响应消息专用头部->只能出现在响应头中
							Server:xxxx	服务器表明自己是什么软件及版本等信息
							Age:xxx		当代理服务器用自己缓存的实体去响应请求时，用该头部表明该实体从产生到现在经过多长时间了。

						响应主体可用头部->描述响应主体特征
							Expires		WEB服务器表明该实体将在什么时候过期，对于过期了的对象，只有在跟WEB服务器验证了其有效性后，才能用来响应客户请求。是 HTTP/1.0 的头部。例如：Expires：Sat, 23 May 2009 10:02:12 GMT
							Content-Type	WEB服务器告诉浏览器自己响应的对象的类型
							Content-Length	WEB服务器告诉浏览器自己响应的对象的长度或尺寸
							Last-Modified	WEB服务器认为对象的最后修改时间
							Location		表示客户应当到哪里去提取文档，用于将接收端定位到资源的位置（URL）上。

						扩展头部->很少用

				3.消息主体/响应正文
				
				状态代码有三位数字组成，第一个数字定义了响应的类别，且有五种可能取值：
					1xx：指示信息--表示请求已接收，继续处理
					2xx：成功--表示请求已被成功接收、理解、接受
					3xx：重定向--要完成请求必须进行更进一步的操作
					4xx：客户端错误--请求有语法错误或请求无法实现
					5xx：服务器端错误--服务器未能实现合法的请求
				常见的响应状态码：
					100 Continue			请求数据以被服务器开始接收，请客户端继续传
					200 OK				响应正常结束，若有响应数据，则在后续的响应正文中给出
					201 Created  			PUT请求提交的数据已经保存完成
					301 Moved Permanently	 	请求的资源已被永久的转移了，往往与Location响应头部配合使用，告诉客户端到另一个URL查找指定的资源
					303 See Other  			与Location响应头部配合使用，告诉客户端参见另一个URL
					304 Not Modified  		客户端此次请求的资源未被修改
					400 Bad Request  		请求消息格式错误
					403 Forbidden 			请求的资源不被允许访问
					404 Not Found 			请求的资源不存在
					405 Method Not Allowed  	请求方法服务器不支持
					500 Server Interval Error 	服务器执行过程中发生错误
					501 Not Implemented 		请求方法不被当前的资源允许
					503 Service Unavailable 	服务暂时不可用
				  	505 HTTP Version Not Supported	HTTP版本不支持

				与缓存控制相关的响应头
					Cache-Control:  max-age=3600		//告诉客户端缓存此次响应数据的时长
					Cache-Control:  max-age=0
					Cache-Control:  no-cache
					Cache-Control:  must-revalidate		//告诉客户端在提供缓存之前必须跟服务器重新验证一次
					Pragma: no-cache
					Expires: Fri,  15 March 2015 12:12:22 GMT	//指定响应数据的确切过期时间
					Expires: 过去的时间/0				//告诉客户端不要缓存响应数据

				项目中控制响应(并非请求,服务器发送实际文档的时候，传给浏览器的信息)消息头的方法：
					(1)	修改Web服务器的配置文件，会影响到其中所有的虚拟主机的所有资源
					(2)	前端工程师：可以修改单个HTML文件的meta标签来控制响应头
						meta http-equiv='Cache-Control'  content='no-cache' 
						meta http-equiv='Expires' content='0'
						meta http-equiv='Pragma' content='no-cache'
						meta http-equiv='Content-Type' content='text/html;charset=UTF-8'
					(3)	后端工程师：可以直接使用代码控制某个资源的响应头
					PHP:  
					JSP:   <%  response.setHeader(‘Expires’,  ‘0’);  %>
					这种控制HTTP响应头的方式优先级要高于使用meta http-equiv=””指定的值

		Web应用中的状态保持
			状态保持：服务器可以得知当前客户端之前曾经的访问历史
			HTTP协议本身是“Stateless”——无状态的。
			HTTP/1.1是一个基于请求/响应模型的、无状态的、面向持久连接的一个TCP/IP协议栈中应用层协议。
			  为了实现“状态保持”功能，有三种技术可以使用：
				(1)	Cookie：把数据存储于客户端磁盘中——是不可靠的状态保持
				(2)	HTML5 Local Storage：把数据存储于客户端磁盘中
				(3)	Session：把数据存储在服务器端——是可靠的状态保持技术

		HTTP和HTTPS
			HTTP：80端口，请求和响应消息都是明文传输，可能被窃听、篡改、伪造
			HTTPS： HTTP over SSL/TLS，443端口，请求和响应消息都是加密传输的
